core.ko是个驱动模块文件，启动脚本里启动了它，利用时就是用的它。

惯常checksec加ida

init_module() 注册了 /proc/core

exit_core() 删除 /proc/core

core_ioctl() 定义了三条命令，分别调用 core_read()，core_copy_func() 和设置全局变量 off

core_read() 从 v4[off] 拷贝 64 个字节到用户空间，但要注意的是全局变量 off 使我们能够控制的，因此可以合理的控制 off 来 leak canary 和一些地址

core_copy_func() 从全局变量 name 中拷贝数据到局部变量中，长度是由我们指定的，这个地方的cp长度是个很明显的有符号到无符号的漏洞

core_write() 向全局变量 name 上写，这样通过 core_write() 和 core_copy_func() 就可以控制 ropchain 了

思路：

由于这种题溢出的是内核程序的栈，所以提权后还是要回到用户模式的。

用户态在转向内核态时，需要保存用户态的一些信息。所以自己写的程序就要用内嵌汇编来保存好需要的寄存器，之后再写到溢出里去。本着拿来主义的原则，save_status之类的cp用就完了。

CISCN2017 - babydriver

还是学习的ctfwiki

看init文件中insmod了babydriver.ko，基本确定搞它

通过释放一块与cred 结构体相同大小的块，利用打开设备两次，free了之后，执行fork，由于fork会复制一个新进程，必定会要一个cred的空间，所以拿到了free的块，通过对另一个设备的更改，完成uaf，设置其的权限。

内核内存分配这块SLAB、SLUB。之前看一些博客了解到slub的分配是LIFO，通过管理cache来分配，每个cache对应一个2的幂次大小的一组内存对象。

open它两次这里的背后原理有些不太理解，它的页表是如何分配的。全局变量可以共享…印象里多线程的分配方式就是全局共享，自己管自己堆栈。留坑。

CISCN2019 babypwn

拿来学习ret2_dl_runtime_resolve

这个利用主要是针对没有write/puts无法使用ret2libc的

核心还是利用dlruntime_resolve(唤醒了我不好的回忆，又是被那个负责人拷问的东西)

当时只是对pltgot学习，知道是通过它来搞的，这次稍微细一点。

dlruntime_resolve调用过程

  1. 用link_map访问.dynamic，取出.dynstr, .dynsym, .rel.plt的指针
  2. .rel.plt + 第二个参数（这个东西就是plt中push的）求出当前函数的重定位表项Elf32_Rel的指针，记作rel
  3. rel->r_info >> 8作为.dynsym的下标，求出当前函数的符号表项Elf32_Sym的指针，记作sym
  4. .dynstr + sym->st_name得出符号名字符串指针
  5. 在动态链接库查找这个函数的地址，并且把地址赋值给*rel->r_offset，即GOT表
  6. 调用这个函数

所以利用方法就是直接调用plt0，自己写个第二个参数，从而控制rel指向，伪造Elf32_Rel，控制.dynsym的下标，伪造

.dynsym项，其name字符串指针指向自己写的”system“，导致后面写入的got变成了system的，于是提前控制好第一个参数为binsh就可以完成了。

0ctf_2018_heapstorm2

一般堆题保护都无所谓了…

分析：

1.上来mmap，在0x13370800保存chunk的大小和指针(0x10) ，0，1位置被提前占用，放了随机数。实际保存是从2开始，2+n。

他们的指针和大小都与0位置的随机数xor，不过由于xor的特性，不影响。

2 .经典菜单，不过view有限制a1[1].m_size ^ a1[1].m_heap) == 322401073才能展示，所以后面需要留神构造。

利用方法：

核心为3个

1.shrink overlapping

以前看overlapping一般都是在change或add时有一个offbyone，但是这个题由于

offbyone在strcpy中（最后补0），前面是一堆乱七八糟的，所以无法随心所欲更改pre_size，因此使用一种shrink overlapping的思路

假设有三个chunk：0，1，2，目的将2和1overlapping，1的size为0x410，0x420之类的，free1后，通过0的offbyone来shrink1，由于1减小，此时修改就不会对2产生影响，之后对free2就可以完成overlapping了

对应本题最开始步骤：

因为overlapping，现在idx7就能控制一个free的unsorted bin，chunk大小为0x4f0，Idx8能控制一个free的large bin

2.利用unsorted bin和large bin的bk和bk_nexsize.

编辑idx7，把unsorted bin的bk改为0x13370800-0x20（0x133707e0），编辑idx8，把large bin的bk改为0x13370800-0x18，bk_nextsize改为0x13370800-0x20-0x18-5，这个unsorted bin的要比largebin的大。
这时，申请一个0x48大小的堆，看运气成功，此时申请出的堆指向0x133707f0。

原理：

malloc(0x48)

首先从unsorted bin里找，找提前构造好的unsorted bin，由于!=,其要放到largebin中，根据源码，由于要放入的比largebin大，所以插入fwd与fwd->bk中间：

victim->bk_nextsize = fwd->bk_nextsize; victim->bk_nextsize->fd_nextsize = victim;

fwd->bk_nextsize为0x133707c3，他的fd_nextsize就为0x133707e3，libc会向其中写入victim的地址。而这个地址的第六字节正好一般是5x开头，所以后面根据错位，0x133707e8为0x5x，所以就正好构造好了。

这里还得碰运气，由于是mmap产生的，还要绕

不过我是真没找到这个检测在哪里…这个mmap检测的就是后3bit的倒数第二个，为1就ok了，所以要赌6.

由于之后malloc(0x48)，所以获得了控制指针和的大小的能力

3.类似于unlink后的操作，修改指针，达到任意写的目的

这里0x13377331就是为了show时绕过a1[1].m_size ^ a1[1].m_heap) != 322401073准备的
